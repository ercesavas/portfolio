Of 1,222 fraud cases in the test set, the Random Forest caught only 190 while missing 1,032. It did keep false positives low (47), which matters for customer experience.
The core tension: Increasing recall (catching more fraud) means lowering the decision threshold, which increases false positives (flagging legitimate transactions). The right tradeoff depends on the business cost of each error type. Missing fraud costs the company money. False-flagging a customer costs you the customer.
What this project doesn't do yet (and should): Threshold tuning: sweeping the decision threshold from 0.5 down to find the operating point where fraud recall vs false positive rate is acceptable. Also SMOTE or class-weight adjustments to directly address the imbalance during training.
04 · Next Steps

What This Project Needs

→
Threshold Optimisation
Default predict() uses a 0.5 threshold. Sweeping from 0.1 to 0.9 and plotting the precision-recall curve would reveal the operating point that best fits the cost function. For fraud detection, you'd likely accept more false positives in exchange for catching more actual fraud.
→
Class Imbalance Handling (SMOTE / class_weight)
With ~5% fraud, the model is biased toward predicting "legitimate" because that's almost always right. SMOTE oversamples the fraud class synthetically; class_weight="balanced" in sklearn reweights the loss function so fraud errors matter more during training. Both would likely double fraud recall.
→
XGBoost / LightGBM
Gradient boosting models consistently outperform Random Forest on tabular imbalanced data. LightGBM in particular has a scale_pos_weight parameter designed exactly for this, and trains in seconds rather than 16 minutes on 1.47M rows.
→
SHAP Values for Explainability
For any real fraud system, you need to explain why a transaction was flagged. SHAP values decompose each prediction into per-feature contributions. Instead of "this transaction is probably fraud," you can say "flagged because: Transaction Amount is 3.2 standard deviations above average and Account Age is only 12 days."

The Most Important Step: Temporal Alignment
Look-ahead bias is how most financial ML projects silently fail. If you use a headline published at 2pm to predict that same day's open-to-close return, the price has already been moving for 4.5 hours, meaning your model is learning from the future. The trading_date() function is the entire reason the results here are honest.
def trading_date(ts):
    t = ts.time()
    d = ts.date()
    if t < dt.time(9, 30): # Pre-market → news applies to TODAY's open-to-close
        return d
    elif t >= dt.time(16, 0): # Post-close → news applies to NEXT DAY's move
        return d + dt.timedelta(days=1)
    else: # Intraday → excluded entirely (price already reacting)
        return None
Why three zones, not two
Most tutorials simply split by date, ignoring the timestamp. But a headline published at 10am on Monday has a completely different relationship to price than one published at 6am Monday or 5pm Monday.

Pre-market (before 9:30am): the market hasn't opened yet. This news genuinely informs the day's move.

Post-close (after 4pm): today's game is done. This news maps to the next trading day's return.

Intraday (9:30am–4pm): the price is already moving in response to everything including this article. Using it as a predictor of today's return is circular. Excluded entirely.
What survives after merging
Starting with 2,770 articles and 1.13M price rows sounds like a lot of data. But after temporal filtering (intraday articles dropped) and the inner join on symbol+date (most price rows have no corresponding news), only 1,474 rows survive into the modelling dataset.

This is the honest cost of doing it correctly. You lose the majority of your data to avoid leakage. The alternative (using all articles regardless of timestamp) would produce better-looking metrics that are completely meaningless.

Why chronological split, not random: A random train/test split on time-series data leaks the future into training. If a 2024 row ends up in the train set, the model learns from data that comes after what it's predicting. Splitting at January 1 2024 ensures the model never sees any 2024 information during training.
# The complete model — two lines
model = Pipeline([
    ("tfidf", TfidfVectorizer(
        max_features=20000, # cap vocabulary at 20k most informative terms
        ngram_range=(1, 2), # unigrams + bigrams ("raises guidance", "misses earnings")
        min_df=2 # ignore terms that appear only once (noise)
    )),
    ("ridge", Ridge(alpha=1.0)) # L2 regularisation — critical for sparse TF-IDF
])

# Target: intraday return
prices["y"] = (prices["close"] - prices["open"]) / prices["open"]
Why TF-IDF
TF-IDF weights terms by how often they appear in a document vs how common they are across all documents. In financial news, "the" is useless because it appears everywhere. But "FDA approval," "earnings beat," or "raises guidance" are rare and informative, and TF-IDF amplifies exactly those.

Bigrams (ngram_range=(1,2)) capture phrases like "misses earnings" vs "beats earnings": the difference between the two words is everything, but unigrams can't see it.
Why Ridge over alternatives
TF-IDF produces a very wide, very sparse matrix: 20,000 columns for 1,474 rows. Ordinary least squares would overfit catastrophically. Ridge's L2 penalty shrinks all coefficients toward zero, penalising complexity and preventing the model from memorising quirks of 2021–2023 news that don't generalise to 2024.

This shrinkage has a direct consequence in the results: the model's predictions have a much tighter spread than actual returns, hedging rather than committing to extreme calls.

04 · Honest Assessment

What This Is and Isn't

What works
✓
Temporal alignment is rigorous
The three-zone timestamp logic is the thing most similar projects get wrong. Here it's explicit and correct.
✓
Chronological train/test split
2024 data held out entirely during training. No leakage from future into past.
✓
Baseline comparison is honest
RMSE compared against "always predict zero", not against a previous run or a held-out random guess.
What's limited
!
1,474 rows is a small dataset
After filtering and merging, only 1,474 matched rows remain. Any model will struggle to generalise robustly from this.
!
TF-IDF doesn't understand negation
"Not profitable" and "profitable" tokenise differently but share most vocabulary. FinBERT would handle this properly.
!
No market regime control
2021–2024 spans bull, bear, and volatile regimes. Patterns learned in 2021's bull market may not hold in 2022's bear.
What Would Make This Better
→
FinBERT for sentiment
A BERT model fine-tuned on financial text would understand "stock plunges on earnings miss" vs "stock rallies on earnings miss", something TF-IDF can't do. The headline "Company fails to raise guidance" reads positive to TF-IDF (sees "raise guidance") but is clearly negative.
→
Body text as additional signal
Cell 20 of the notebook starts this by concatenating headline and article body. The body contains specific numbers (EPS, revenue, guidance) that are far more predictive than the headline alone. Already scaffolded, just needs evaluation.
→
Sector/market-cap grouping
The top-10 predictions are dominated by small-cap biotech, a sector where a single binary event (FDA approval/rejection, financing announcement) fully determines price. Separating by sector and training separate models would likely sharpen predictions significantly.
→
Directional classifier instead of regressor
Since the model's main value is direction (56.97% accuracy), reframing as a three-class problem (up/flat/down) with a classification loss would directly optimise the thing that matters. Ridge regression optimises MSE of the return magnitude, which is not the same objective.
